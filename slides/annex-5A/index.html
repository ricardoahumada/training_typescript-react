1. Contexto: El problema del cliente

10 minutos

Escenario: "Se renderiza en el back"

El cliente dice esto pero usualmente significa que usa Webpack, que solo empaqueta JS. SSR real necesita un framework que ejecute React en el servidor.

La confusión común:

  • "Uso Webpack" → pero Webpack solo empaqueta, NO hace SSR
  • "Quiero renderizar en backend" → necesitas un FRAMEWORK

Lo que el cliente ACTUALMENTE tiene:

Webpack → empaqueta JS → navegador → React hydrate → render (NO es SSR)

Webpack != SSR

Webpack es un bundler, no un framework. Solo empaqueta archivos, no ejecuta React en servidor.

4. Server vs Client Components

15 minutos

La regla de oro

Server Components por defecto. Solo usar 'use client' cuando necesites hooks o eventos.

En Next.js App Router, los componentes son Server Components por defecto.

CaracterísticaServerClient
RenderizadoServidorCliente
Hooks (useState, useEffect)NoSí
Event handlersNoSí
Acceso BD/FSSíNo

Árbol de decisiones

Pregúntate: ¿tiene interactividad? ¿accede a BD/FS? Responde las preguntas para saber qué tipo de componente usar.

¿Tiene interactividad? (useState, onClick)
  → SI → Client Component ('use client')
  → NO → ¿Accede a BD/FS?
      → SI → Server Component
      → NO → Server Component (por defecto)

Server vs Client: Beneficios Concretos

Server: SEO automático, acceso directo a BD, 0KB JS. Client: interactividad completa.

AspectoServerClient
SEO✅ Automatico❌ Requiere cliente
Acceso BD/FS✅ Directo❌ Via API
Bundle size✅ 0 KB JS❌ Todo el codigo
Interactividad❌ No✅ Completa
Time to interactive✅ Inmediato❌ Espera JS

Tip: Usa Server por defecto. Solo cambia a Client cuando NECESITES hooks o eventos.

  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}

5. Routing y API Routes

15 minutos

Routing: app/tasks/[id]/page.tsx

El App Router usa archivos para definir rutas. Los parámetros vienen en `params` (promesa en Next.js 15+).

interface PageParams {
  params: { id: string };
}

export default async function TaskPage({ params }: PageParams) {
  const task = await getTask(params.id);
  return <TaskDetail task={task} />;
}

Route Handlers: app/api/tasks/route.ts

API Routes permiten crear endpoints REST directamente en Next.js.

export async function GET() {
  const tasks = await getTasks();
  return NextResponse.json(tasks);
}

export async function POST(request: NextRequest) {
  const body = await request.json();
  const created = await createTask(body);
  return NextResponse.json(created, { status: 201 });
}

Pages vs API Routes: Cuándo usar cada uno

Pages para UI (SEO). API Routes para lógica (CRUD, auth, webhooks).

AspectoPagesAPI Routes
PropositoRenderizar UIManejar requests
ReturnsHTML/JSXJSON
BrowserDirectoVia fetch
SEOIndexableNo indexable

Regla: Pages para UI. API Routes para logic (CRUD, auth, webhooks).

6. Tipado de Server Components

15 minutos

Params como Promise (Next.js 15+)

En Next.js 15+, `params` es una Promise que debe resolverse con `await`.

interface PageProps {
  params: Promise<{ id: string }>;
}

export default async function TaskPage({ params }: PageProps) {
  const { id } = await params;
  const task = await getTask(id);
  return <TaskDetail task={task} />;
}

SearchParams tipado

searchParams también es Promise. Tipalo para evitar errores en runtime.

interface PageSearchParams {
  searchParams: Promise<{ page?: string; filter?: string }>;
}

export default async function TasksPage({ searchParams }: PageSearchParams) {
  const { page, filter } = await searchParams;
  // ...
}

Tipado: Por qué importa en Server Components

Los tipos dan seguridad en refactoring, autocompletado en el IDE, y errores en build no en producción.

Sin tiposCon tipos
params.id puede ser string | undefinedparams.id es string
task.title puede fallar en runtimetask.title es string
Refactoring peligrosoRefactoring seguro
searchParams sin autocompletesearchParams con autocomplete

Beneficio: Errores en build, no en production. DX mejorado con autocomplete.

7. Data Fetching con Tipos

15 minutos

Fetch tipado + Zod

Un solo schema Zod define tipos y validación. Útil para compartir tipos entre frontend y backend.

import { z } from 'zod';

const TaskSchema = z.object({
  id: z.number(),
  title: z.string(),
  status: z.enum(['pending', 'completed']),
});

type Task = z.infer<typeof TaskSchema>;

async function getTasks(): Promise<Task[]> {
  const res = await fetch('/api/tasks');
  return res.json();
}

Beneficio: Un solo schema = tipos en frontend y backend

Fetch + Zod: Por qué mejor que any

// ❌ Con any - sin validación, sin tipos
const data: any = await fetch('/api/tasks').then(r => r.json());
data.title.toUpperCase(); // Puede fallar en runtime!

// ✅ Con Zod - validación + tipos
const data = await fetch('/api/tasks')
  .then(r => r.json())
  .then(data => TaskSchema.array().parse(data));
data[0].title.toUpperCase(); // Type-safe!
AspectoanyZod
Validación❌ No✅ Sí
ErroresRuntimeBuild time
Autocomplete❌ No✅ Sí

Lab: Data Fetching Tipado con Zod

// types/task.ts
import { z } from 'zod';

export const TaskSchema = z.object({
  id: z.number(),
  title: z.string().min(1).max(100),
  status: z.enum(['pending', 'inProgress', 'completed']),
  priority: z.enum(['low', 'medium', 'high']),
});

export type Task = z.infer<typeof TaskSchema>;
export type CreateTaskInput = z.infer<typeof CreateTaskSchema>;

// lib/api.ts
export async function getTasks() {
  const res = await fetch('/api/tasks');
  const data = await res.json();
  return TaskSchema.array().parse(data); // throw if invalid
}

export async function createTask(input: CreateTaskInput) {
  const res = await fetch('/api/tasks', {
    method: 'POST',
    body: JSON.stringify(input),
  });
  return TaskSchema.parse(await res.json());
}

8. Streaming y Suspense

15 minutos

Archivos especiales

Next.js reconoce estos archivos automáticamente: loading.tsx, error.tsx, not-found.tsx.

ArchivoCuando se muestra
loading.tsxMientras carga la página
error.tsxCuando hay error
not-found.tsxCuando notFound() es llamado

Suspense

Suspense permite mostrar contenido de fallback mientras los componentes asíncronos cargan.

import { Suspense } from 'react';

export default function Page() {
  return (
    <Suspense fallback={<Skeleton />}>
      <TaskList />
    </Suspense>
  );
}

Streaming: Mejora UX en aplicaciones reales

El contenido aparece progresivamente en lugar de esperar a que todo cargue. Reduce la tasa de rebote.

Sin StreamingCon Streaming
Pagina en blanco hasta que TODO cargueContenido aparece progresivamente
Usuario espera 3-5s sin feedbackSkeleton muestra progreso
Alta tasa de reboteUsuario ve contenido parcial antes

Caso real: E-commerce con 100 productos. Sin streaming: 5s de espera. Con streaming: productos aparecen en 500ms.

Headers, Cookies y Query Params

Las API Routes acceden a headers, cookies y query params a través del objeto NextRequest.

export async function GET(request: NextRequest) {
  const authHeader = request.headers.get('authorization');
  const sessionCookie = request.cookies.get('session');
  const { searchParams } = new URL(request.url);
  const page = searchParams.get('page');
  
  return NextResponse.json({ auth, session, page });
}

Lab: API Routes Completos

// app/api/tasks/route.ts
export async function GET(request: NextRequest) {
  const { searchParams } = new URL(request.url);
  const status = searchParams.get('status');
  const page = Number(searchParams.get('page') || 1);

  const tasks = await getTasks({ status, page });
  return NextResponse.json(tasks);
}

export async function POST(request: NextRequest) {
  const body = await request.json();
  const result = CreateTaskSchema.safeParse(body);

  if (!result.success) {
    return NextResponse.json({ error: result.error }, { status: 400 });
  }

  const task = await createTask(result.data);
  return NextResponse.json(task, { status: 201 });
}

// app/api/tasks/[id]/route.ts
export async function GET(request: NextRequest, { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const task = await getTask(id);

  if (!task) return NextResponse.json({ error: 'Not found' }, { status: 404 });
  return NextResponse.json(task);
}

export async function PATCH(request: NextRequest, { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const body = await request.json();
  const updated = await updateTask(id, body);
  return NextResponse.json(updated);
}

export async function DELETE(request: NextRequest, { params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  await deleteTask(id);
  return new NextResponse(null, { status: 204 });
}
