4. Server vs Client Components

15 minutos

La regla de oro

En Next.js App Router, los componentes son Server Components por defecto.

CaracterísticaServerClient
RenderizadoServidorCliente
Hooks (useState, useEffect)NoSí
Event handlersNoSí
Acceso BD/FSSíNo

Árbol de decisiones

¿Tiene interactividad? (useState, onClick)
  → SI → Client Component ('use client')
  → NO → ¿Accede a BD/FS?
      → SI → Server Component
      → NO → Server Component (por defecto)

Lab: Migrar componente a Server Component

Objetivo: Convertir un componente de SPA en Server Component

  1. Analizar qué necesita el componente
  2. Si no tiene interactividad → Server Component
  3. Si tiene useState/useEffect → 'use client'

7. Data Fetching con Tipos

15 minutos

Fetch tipado + Zod

import { z } from 'zod';

const TaskSchema = z.object({
  id: z.number(),
  title: z.string(),
  status: z.enum(['pending', 'completed']),
});

type Task = z.infer<typeof TaskSchema>;

async function getTasks(): Promise<Task[]> {
  const res = await fetch('/api/tasks');
  return res.json();
}

Beneficio: Un solo schema = tipos en frontend y backend

Metadata dinámica

export async function generateMetadata({ params }: Props): Promise<Metadata> {
  const { id } = await params;
  const task = await getTask(id);
  
  return {
    title: task ? `${task.title} - Task Manager` : 'Not found',
  };
}
