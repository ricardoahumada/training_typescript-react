El problema de los tipos en runtime

Los datos de APIs externas no tienen tipos en tiempo de ejecución

Problema: TypeScript solo valida en compilación

2. Type Guards

Validación de tipos en runtime

¿Qué son los Type Guards?

Funciones que validan el tipo de un valor en tiempo de ejecución

  • Permiten validar el tipo en runtime
  • TypeScript usa esta información para narrowing
  • Esenciales para aplicaciones que consumen datos externos

Type Guard con typeof

function procesarDato(dato: string | number): string {
  if (typeof dato === 'string') {
    return dato.toUpperCase(); // TypeScript sabe que es string
  }
  return dato.toFixed(2); // TypeScript sabe que es number
}

Type Guard con instanceof

class Rectangulo {
  constructor(public ancho: number, public alto: number) {}
  area() { return this.ancho * this.alto; }
}

class Circulo {
  constructor(public radio: number) {}
  area() { return Math.PI * this.radio ** 2; }
}

type Figura = Rectangulo | Circulo;

function calcularArea(figura: Figura): number {
  if (figura instanceof Rectangulo) {
    return figura.area();
  }
  return figura.area();
}

Type Guard Personalizado

type Persona = Usuario | Administrador;

function isAdministrador(persona: Persona): persona is Administrador {
  return 'permisos' in persona && persona.tipo === 'admin';
}

function procesarPersona(persona: Persona) {
  if (isAdministrador(persona)) {
    console.log('Admin con permisos:', persona.permisos);
  } else {
    console.log('Usuario:', persona.nombre);
  }
}

  • Evita errores de tipo al acceder al contexto

5. Discriminated Unions

State Machines Type-Safe

¿Qué son las Discriminated Unions?

Patrón fundamental que usa una propiedad común (discriminador) para identificar la variante

  • TypeScript infiere automáticamente el tipo según el discriminador
  • Permite exhaustiveness checking
  • Ideales para reducers y gestión de estado compleja

Estructura Básica

type TaskManagerAction =
  | { type: 'ADD_TASK'; payload: Task }
  | { type: 'REMOVE_TASK'; payload: string }
  | { type: 'UPDATE_TASK'; payload: Task }
  | { type: 'SET_TASKS'; payload: Task[] }
  | { type: 'ADD_TAG'; payload: Tag }
  | { type: 'SET_TAGS'; payload: Tag[] }
  | { type: 'CLEAR_ERROR' };

Reducer con Type Safety

function taskReducer(state: TaskManagerState, action: TaskManagerAction): TaskManagerState {
  switch (action.type) {
    case 'ADD_TASK':
      // TypeScript sabe que action.payload es Task
      return { ...state, tasks: [...state.tasks, action.payload] };
    
    case 'REMOVE_TASK':
      // TypeScript sabe que action.payload es string
      return { ...state, tasks: state.tasks.filter(t => t.id !== action.payload) };
    
    // ...
    
    default:
      // Exhaustiveness checking
      const _exhaustive: never = action;
      return state;
  }
}

Ventaja: Si falta un caso, error de compilación

6. Componentes Avanzados

Polimórficos, Compound y forwardRef

Tipos de Componentes Avanzados

  • Componentes polimórficos: mismo componente puede renderizar como button, link, etc.
  • Compound components: API declarativa donde componentes hijos se comunican con el padre
  • forwardRef: acceder a elementos del DOM

Componente Polimórfico

type PolymorphicRef<C extends React.ElementType> =
  React.ComponentPropsWithRef<C>['ref'];

interface ComponentePolimorficoProps<T extends React.ElementType = 'button'> {
  as?: T;
  children: React.ReactNode;
  ref?: PolymorphicRef<T>;
}

const ComponentePolimorfico = React.forwardRef(
  <T extends React.ElementType = 'button'>(
    { as, children, ...props }: ComponentePolimorficoProps<T>,
    ref: PolymorphicRef<T>
  ) => {
    const Component = as || 'button';
    return <Component ref={ref} {...props}>{children}</Component>;
  }
);

Compound Components

function TaskManager({ children }: TaskManagerCompoundProps) {
  return <div className="task-manager">{children}</div>;
}

function TaskManagerItem({ children, value }: TaskManagerItemProps) {
  return <div data-value={value}>{children}</div>;
}

TaskManager.Item = TaskManagerItem;

7. Validación Runtime con Zod

Esquemas con inferencia automática de tipos

¿Qué es Zod?

Librería de validación de esquemas que permite validar datos en runtime e inferir tipos para TypeScript

Problema resuelto: TypeScript no valida en ejecución

Esquemas con Inferencia

import { z } from 'zod';

const LibroSchema = z.object({

8. Custom Hooks Tipados

Hooks personalizados con tipos estrictos

¿Por qué tipar los hooks?

  • Proporcionan autocompletado al usar el hook
  • Definen claramente qué datos devuelve el hook
  • Validan que se usan correctamente
  • Permiten crear hooks genéricos reutilizables

useState y useEffect con Tipos

// Tipado explícito
const [usuario, setUsuario] = useState<Usuario | null>(null);

// Tipado con genérico
const [items, setItems] = useState<Libro[]>([]);

function useCargarLibros() {
  const [libros, setLibros] = useState<Libro[]>([]);
  const [cargando, setCargando] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    async function fetchLibros() {
      try {
        const response = await fetch('/api/libros');
        const data = await response.json();
        setLibros(data);
      } catch (e) {
        setError(e as Error);
      } finally {
        setCargando(false);
      }
    }
    fetchLibros();
  }, []);

  return { libros, cargando, error };
}

Hooks Genéricos

function useColeccion<T extends { id: string }>(endpoint: string) {
  const [items, setItems] = useState<T[]>([]);
  const [cargando, setCargando] = useState(false);

  const agregar = useCallback((item: T) => {
    setItems(prev => [...prev, item]);
  }, []);

  const eliminar = useCallback((id: string) => {
    setItems(prev => prev.filter(item => item.id !== id));
  }, []);

  return { items, cargando, agregar, eliminar };
}

// Uso con tipo específico
const { items: libros, agregar, eliminar } = useColeccion<Libro>('/api/libros');

9. React 18 Concurrency

Mantener la UI responsiva

¿Por qué usar Concurrencia?

React 18 introdujo capacidades que permiten trabajar en múltiples tareas simultáneamente

  • useTransition: marca actualizaciones como "no urgentes"
  • useDeferredValue: retrasa valores para evitar bloqueos
  • Suspense: permite mostrar estados de carga controlados

useTransition

function BusquedaTareas() {
  const [busqueda, setBusqueda] = useState('');
  const [resultados, setResultados] = useState<Task[]>([]);
  
  const [isPending, startTransition] = useTransition();

  const handleBuscar = (query: string) => {
    setBusqueda(query);
    
    // La búsqueda se marca como transición
    startTransition(() => {
      const filtrados = tasks.filter(task => 
        task.title.toLowerCase().includes(query.toLowerCase())
      );
      setResultados(filtrados);
    });
  };

  return (
    <div>
      <input value={busqueda} onChange={(e) => handleBuscar(e.target.value)} />
      {isPending && <Spinner />}
      <ListaTareas tasks={resultados} />
    </div>
  );
}

useDeferredValue

function ListaFiltrada({ items }: { items: Task[] }) {
  const [filtro, setFiltro] = useState('');
  
  // DeferredValue retrasa el valor
  const filtroDiferido = useDeferredValue(filtro);
  
  const itemsFiltrados = useMemo(
    () => items.filter(item => 
      item.title.includes(filtroDiferido)
    ),
    [items, filtroDiferido]
  );

  return (
    <div>
      <input value={filtro} onChange={e => setFiltro(e.target.value)} />
      <ul>{itemsFiltrados.map(item => (
        <li key={item.id}>{item.title}</li>
      ))}</ul>
    </div>
  );
}

Suspense para Data Fetching

10. Tipos en Next.js App Router

Server y Client Components

Server vs Client Components

  • Server Components: por defecto, pueden ser async, fetching directo a DB
  • Client Components: necesitan directiva 'use client', usan hooks y estado
  • Las props de pages y API routes tienen tipos específicos
  • Server Actions requieren tipado para validación

Server Component

// app/tasks/page.tsx - Server Component
interface PageProps {
  params: Promise<{ estado?: string }>;
  searchParams: Promise<{ q?: string }>;
}

export default async function PaginaTareas({
  params, 
  searchParams

  const { estado } = await params;
  const { q } = await searchParams;

  return (

12. Mapped Types & Conditional Types

Transformación dinámica de tipos

¿Por qué usarlos?

  • Mapped Types: transformar propiedades de un tipo
  • Conditional Types: crear tipos que dependen de otros tipos
  • Crear utility types personalizados
  • Validar y transformar tipos en tiempo de compilación

Mapped Types

// Transformar todas las propiedades a opcionales
type Partial<T> = {
  [P in keyof T]?: T[P];
};

// Hacer todas las fechas strings
type FechasAString<T> = {
  [P in keyof T]: T[P] extends Date ? string : T[P];
};

interface Prestamo {
  id: string;
  fechaInicio: Date;
  fechaFin: Date;
}

type PrestamoSerializado = FechasAString<Prestamo>;
// { id: string; fechaInicio: string; fechaFin: string; }

Conditional Types

// Tipo condicional básico
type IsString<T> = T extends string ? true : false;

// Extraer el tipo de elemento de un array
type ElementType<T> = T extends (infer U)[] ? U : never;

// Obtener tipo de retorno de una función
type ReturnType<T> = T extends (...args: any[]) => infer R ? R : never;

Type-Safe API Responses

type ApiSuccess<T> = {
  success: true;
  data: T;
};

type ApiError = {
  success: false;
  error: string;
  code: number;
};

type ApiResponse<T> = ApiSuccess<T> | ApiError;

function handleResponse<T>(response: ApiResponse<T>) {
  if (response.success) {
    // TypeScript sabe que response.data existe
    console.log(response.data);
  } else {
    // TypeScript sabe que response.error existe
    console.error(response.error, response.code);
  }
}
